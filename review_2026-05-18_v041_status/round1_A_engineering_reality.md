## 1. Verdict

**v0.4.1 is NOT release-ready. Honest readiness: ~75%.** The release is also factually still in progress: `CHANGELOG.md:10` says `[0.4.1] ŌĆö In progress`, the `[Unreleased]` compare link still points to `v0.4.0...HEAD`, and no `v0.4.1` git tag exists ŌĆö so "merged to main" ŌēĀ "released," which is correct hygiene. Static analysis is clean (ruff: 0 findings; pyright: 0 errors, 3 pre-existing benign warnings). The big v0.4.1 surfaces that I spot-verified are mostly correct: L1.3 single-source (verified empirically), L3.1 URI carve-out, L6.1ŌĆōL6.3 PROV split, the L5.x AttributionAudit metrics, and the FM1/FM25 same-instance generalization at the tuple level. **However, the single highest-criticality item ŌĆö the evt_025 attribution guard (FINAL L3.2 + L5.4, Priority 2 "critical") ŌĆö shipped the *rejected* pre-Round-3 design.** Round-3 amendment A1 explicitly required a **multi-actor guard** as the *precondition* for L3.2/L5.4 being deemed "clean"; that guard is entirely absent from merged code, and the test suite was rewritten to assert the buggy behavior rather than enforce A1. This re-introduces the exact false positives A1 was written to prevent (named real production data). Separately, **the reported FM7 regression does not reproduce as a code defect** ŌĆö it passes deterministically here (see ┬¦3); it is an E2E timeout flake under machine load, not a v0.4.1 logic bug. The release blocker is therefore the A1 scoping regression and the test-suite-encodes-the-bug problem, not FM7.

## 3. Root-cause deep-dive ŌĆö the FM7 / `trace_end_session` "regression"

**Conclusion: there is no v0.4.1 code defect causing FM7. It is an E2E timeout flake under host load. The brief's framing ("reproduces in isolation in ~16s, NOT a flake/env") is not reproducible on this machine.**

### Evidence chain

1. **FM7 test file was not touched by v0.4.1.** `git diff 7c9e474..d20be80 -- tests/test_failure_modes_e2e.py` ŌåÆ 0 lines. Last modified in `cba1a86` (pre-v0.4.1). FM7 (`test_failure_modes_e2e.py:285-307`) starts a server subprocess, logs 8 annotations with the helper default `category="learning"` (`_annotate`, line 85-96 ŌĆö no `actor_type`, no `conversation_snippet`), then calls `_end_session` (`summary="test"`).

2. **The exact FM7 logic path executes correctly and instantly in-process.** Probe (via `.venv/bin/python` with `PYTHONPATH=.../src`, importing the *real* `trace_mcp.server` module and calling the actual MCP tool functions):
   - `trace_start_session` ŌåÆ 8├Ś `trace_log_annotation(category="learning", content="Batch annotation i")` ŌåÆ `trace_end_session(summary="test")`.
   - Result: **END OK in 0.001s**, output = `Session ended ŌĆ” 8 events: 8 annotations\n--- Attribution Audit ---\nNo contributions, decisions, or corrections to review.\nContext saved: ŌĆ”`. No exception, no hang.
   - Re-ran the `end_session` + `scratchpad.write_scratchpad` slice separately: also 0.000s, clean.

3. **`_build_attribution_audit` is safe for FM7's input.** Read in full (`session_tools.py:220-435`): for 8 plain `learning` annotations every branch is skipped (no contributions/decisions/corrections; `discovery_anchors` empty; the orphan-discovery scan is O(K┬ĘN) with K=0; L5.6 dispatch hint gated on `contribution_count>=10` ŌåÆ not reached; `env is not None` guard present at line 402 so the A7 env-None concern does **not** apply). No regex runs on `session.summary` (F7). No `None`-deref. No loop over anything unbounded. `timedelta`/`datetime` arithmetic only on real timestamps.

4. **The server wrapper cannot "return no response" from a Python exception.** `server.py:194-245` wraps the whole body in `try/except Exception` ŌåÆ returns `f"Error ending session: {e}"`. `extract_if_available` (`extension_hooks.py:99-116`) is fail-open (its own try/except, never raises). The scratchpad block (`server.py:230-240`) has its own try/except. So a *caught* exception yields a JSON-RPC response with an error string ŌĆö it would **not** produce the observed "no response ŌåÆ readline TimeoutError." Only a hard process stall/no-output produces that.

5. **FM7 passes deterministically here.** Isolation: 4/4 PASS (4.44s / 7.58s / 3.82s / 3.53s). Full `TestAttributionFailures` class: 13/13 PASS (25.18s). `load average` during runs Ōēł 1.8. The brief's "~16s, reproduces in isolation, not a flake" is contradicted by direct measurement.

6. **The actual failure surface is the E2E read timeout.** `tests/test_e2e_server.py:59-96` `_send_and_receive` does `await asyncio.wait_for(proc.stdout.readline(), timeout=15.0)`. The harness **does not capture/print `proc.stderr`** on timeout (`_shutdown_server` at :155-163 just closes stdin / kills). With 8+ concurrent Claude sessions contending for CPU/IO, the subprocess `python -m trace_mcp.server` cold start (import of `mcp`/FastMCP + eager `model2vec` import + embedding-provider instantiation, F5) can exceed 15s on the unlucky run ŌåÆ `TimeoutError`. This is consistent with *every* symptom in the brief without any v0.4.1 logic defect.

### Most plausible latency contributor (F5, pre-existing)

`_load_extensions()` (`server.py:715-758`) imports the learn extension whose `__init__.py:39` calls `get_embedding_provider()` at **startup**; `embeddings.py:40-43` imports `model2vec` at module load, and the model2vec provider's constructor downloads `potion-base-8M` from HuggingFace on a cold cache *before the server answers `initialize`*. On a quiet machine cold init RTT measured 0.43s, but a cold model cache or HF stall + heavy host load can push a single subprocess over the 15s harness ceiling. This vector predates v0.4.1 (learn-dir v0.4.1 diff: `models.py +3`, `embeddings.py 0`).

### Mapping to FINAL-plan / Round-3 items

**None.** The FM7 symptom maps to *no* v0.4.1 plan item. The brief's two candidate mappings are both refuted: (a) "regex on session.summary per A2" ŌĆö A2's sanctioned outcome was *removing* L5.7, which was done; no such regex exists (F7). (b) "`session.metadata.environment is None` per A7" ŌĆö the only environment dereference in the audit (`session_tools.py:399-405`, L5.6) is explicitly `env is not None`-guarded, and is not even reached for FM7 (gated on `contribution_count>=10`).

### Do other code paths share a defect?

No shared *defect* (there is no defect). The shared *risk* is F5's eager startup import/model load, which affects **every** E2E test that starts a server subprocess (not FM7-specific) ŌĆö which is exactly why FM7 looks like a flake rather than a localized failure. The genuine v0.4.1 correctness bug (F1/F2) is in a *different* path (`resolve_decision` / `_build_attribution_audit` same-instance detection) and is unrelated to FM7.
